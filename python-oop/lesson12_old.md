# Урок 12. Метод `__call__`: как сделать объект функцией и зачем это нужно

---

В этом уроке мы разберём один из самых недооценённых, но при этом крайне полезных магических методов — `__call__`. На первый взгляд он кажется простым: «позволяет вызывать объект как функцию». Но за этой формулировкой скрывается гораздо более важная идея — **объект, который умеет хранить состояние, начинает вести себя как функция**.

Прежде чем переходить к примерам и практике, нужно снять одну очень важную путаницу, которая почти всегда возникает у студентов на этом этапе.

---

## Два разных `__call__` в Python

В Python существует **два разных уровня**, на которых используется метод `__call__`.

Первый уровень — это уровень **создания объектов**.

Когда вы пишете:

```python
obj = MyClass()
```

вы фактически **вызываете** класс. И здесь действительно используется метод `__call__`. Но это **не тот метод**, который мы будем изучать в этом уроке.

Второй уровень — это уровень **экземпляра класса**.
Именно этот `__call__` мы будем рассматривать. 

Он отвечает за следующее:

> что должно происходить, когда мы вызываем **уже созданный объект** как функцию

То есть:

```python
obj()   # вот это вызывает __call__ у экземпляра
```

Очень важно зафиксировать это различие:

* `MyClass()` — вызывает механизм создания объекта (другой уровень)
* `obj()` — вызывает `__call__` у объекта

Один и тот же синтаксис, но совершенно разный смысл.

---

## Краткое замечание о метаклассах

Чтобы не оставлять «белое пятно», нужно кратко проговорить, что происходит при создании объектов.

Когда вы вызываете класс:

```python
obj = MyClass()
```

Python выполняет не просто вызов функции. Внутри работает более сложный механизм, связанный с так называемыми **метаклассами** — это «классы для классов», которые определяют, как создаются объекты.

На этом уровне действительно есть свой `__call__`, который:

* вызывает `__new__`
* затем вызывает `__init__`

Но сейчас это для нас избыточная сложность. В рамках этого урока важно лишь понять:

> управление созданием объектов — это отдельный уровень, и мы к нему вернёмся позже (например, при разборе Singleton)

Сейчас нас интересует другой сценарий — когда объект уже создан.

---

## Переход к основной идее: объект как функция

Теперь сформулируем проблему, которую решает `__call__`.

В программировании у нас есть два основных инструмента:

* **функции** — выполняют действия
* **классы** — хранят состояние и структуру

Но иногда возникает ситуация, когда:

* нам нужно **выполнять действие**, как функция
* и одновременно **хранить состояние**, как объект

Функции с этим справляются не всегда удобно. Классы — справляются, но их нельзя просто вызвать.

Возникает разрыв:

```text
Функция → можно вызвать, но нет структуры
Объект → есть структура, но нельзя вызвать
```

Метод `__call__` как раз и закрывает этот разрыв.

> Он позволяет сделать так, чтобы объект вел себя как функция.

---

## Самый простой пример

Пример, который демонстрирует саму механику.

```python
class Counter:
    def __init__(self):
        self.count = 0

    def __call__(self):
        self.count += 1
        return self.count
```

Создадим объект:

```python
c = Counter()
```

Теперь попробуем его вызвать:

```python
print(c())  # 1
print(c())  # 2
print(c())  # 3
```

Что здесь происходит?

* объект `c` хранит состояние (`count`)
* при каждом вызове `c()` вызывается метод `__call__`
* состояние изменяется

Если попытаться описать это максимально просто:

> `__call__` — это метод, который определяет, что делает объект при записи `obj()`

Важно также заметить:

мы нигде явно не вызываем `__call__`. Python делает это автоматически.

---

## Важное сравнение: замыкание vs класс

На этом этапе у вас может возникнуть логичный вопрос:

> “Зачем вообще нужен `__call__`, если можно использовать функции?”

Это очень правильный вопрос. Чтобы ответить на него, сравним два подхода.

---

### Вариант 1. Замыкание (closure)

```python
def create_counter():
    count = 0

    def counter():
        nonlocal count
        count += 1
        return count

    return counter


c = create_counter()

print(c())  # 1
print(c())  # 2
print(c())  # 3
```

Здесь:

* функция хранит состояние (`count`)
* работает корректно

---

### Вариант 2. Класс с `__call__`

```python
class Counter:
    def __init__(self):
        self.count = 0

    def __call__(self):
        self.count += 1
        return self.count


c = Counter()

print(c())  # 1
print(c())  # 2
print(c())  # 3
```

---

### В чём разница?

На первый взгляд — почти никакой. Но это только на простом примере.

Если посмотреть глубже:

**Замыкание:**

* логика «спрятана» внутри функции
* сложно расширять
* трудно добавлять дополнительные методы
* сложно масштабировать

**Класс:**

* структура явно описана
* можно добавить методы (`reset`, `get_count` и т.д.)
* легче читать и поддерживать
* лучше подходит для сложной логики

Например, легко расширить класс:

```python
class Counter:
    def __init__(self):
        self.count = 0

    def __call__(self):
        self.count += 1
        return self.count

    def reset(self):
        self.count = 0
```

С замыканием сделать это гораздо сложнее и менее очевидно.

---

### Вывод

> `__call__` — это способ заменить замыкание более структурированным и масштабируемым решением.

---

## Зачем нужен `__call__`: системное понимание

Теперь, когда мы увидели механику и сравнили с альтернативой, можно сформулировать более общее понимание.

Метод `__call__` нужен не сам по себе. Он решает конкретные задачи.

---

### 1. Хранение состояния между вызовами

Объект может запоминать данные:

```python
c = Counter()
c()
c()
```

И это состояние живёт внутри объекта.

---

### 2. Конфигурируемое поведение

Можно создавать разные объекты с разными настройками:

```python
class Multiplier:
    def __init__(self, factor):
        self.factor = factor

    def __call__(self, x):
        return x * self.factor


double = Multiplier(2)
triple = Multiplier(3)

print(double(10))  # 20
print(triple(10))  # 30
```

---

### 3. Повторное использование логики

Объект можно передавать как функцию:

```python
def process(func, value):
    return func(value)

print(process(double, 5))  # 10
```

---

## `__call__` - Основа для архитектурных решений

Именно через `__call__` реализуются:

* декораторы на основе классов
* кеширование
* логирование
* прокси-объекты
* стратегии обработки

То есть, это уже не просто «синтаксический сахар», а инструмент архитектуры.

---

## Декораторы через класс: от функции к объекту

Ранее вы уже работали с декораторами как с функциями. Напомним базовую конструкцию:

```python
def logger(func):
    def wrapper(*args, **kwargs):
        print("Функция вызвана")
        return func(*args, **kwargs)
    return wrapper
```

И использование:

```python
@logger
def add(a, b):
    return a + b
```

Здесь важно понимать структуру:

* внешняя функция (`logger`) принимает функцию
* внутренняя функция (`wrapper`) выполняет дополнительную логику
* затем вызывается исходная функция

Это классический пример использования **замыкания**.

Теперь обратим внимание на ключевой момент:

внутри декоратора нам нужно:

1. сохранить функцию
2. обернуть её выполнение

Ровно это же мы можем сделать с помощью класса.

---

### Перепишем декоратор через класс

```python
class Logger:
    def __init__(self, func):
        self.func = func  # сохраняем функцию

    def __call__(self, *args, **kwargs):
        print("Функция вызвана")
        return self.func(*args, **kwargs)
```

Использование остаётся тем же:

```python
@Logger
def add(a, b):
    return a + b

print(add(2, 3))
```

Разберём, что здесь происходит.

Когда Python видит:

```python
@Logger
def add(a, b):
    ...
```

он фактически выполняет:

```python
add = Logger(add)
```

То есть:

1. создаётся объект класса `Logger`
2. в `__init__` передаётся функция `add`
3. переменная `add` теперь ссылается **не на функцию**, а на объект

А дальше происходит самое важное:

```python
add(2, 3)
```

Python пытается «вызвать» объект.

И автоматически вызывает:

```python
Logger.__call__(add_object, 2, 3)
```

Таким образом:

* `__init__` → отвечает за настройку (сохранение функции)
* `__call__` → отвечает за выполнение

Именно поэтому формулировка:

> класс-декоратор = объект + логика вызова

**является точной и фундаментальной**.

---

## Почему это лучше, чем функция?

Как только логика такого декоратора усложняется:

* появляется состояние
* нужно хранить параметры
* нужно расширять поведение

класс начинает выигрывать у функции.

---

## **Декораторы с параметрами**: усложнение структуры

Ранее вы уже писали декораторы с параметрами:

```python
def logger(level):
    def decorator(func):
        def wrapper(*args, **kwargs):
            print(f"[{level}] Вызов функции")
            return func(*args, **kwargs)
        return wrapper
    return decorator
```

Использование:

```python
@logger(level="INFO")
def add(a, b):
    return a + b
```

Здесь структура становится трёхуровневой:

```text
logger → decorator → wrapper
```

Теперь посмотрим, как это реализуется через класс.

---

### Класс-декоратор с параметрами

```python
class Logger:
    def __init__(self, level):
        self.level = level  # сохраняем параметр

    def __call__(self, func):
        # этот вызов происходит при декорировании
        def wrapper(*args, **kwargs):
            print(f"[{self.level}] Вызов функции")
            return func(*args, **kwargs)
        return wrapper
```

Использование:

```python
@Logger(level="INFO")
def add(a, b):
    return a + b
```

Теперь разберём, что происходит по шагам:

1. `Logger(level="INFO")` → создаётся объект
2. этот объект вызывается с функцией `add`
3. `__call__` возвращает `wrapper`
4. `add` теперь указывает на `wrapper`

Обратите внимание:
здесь `__call__` вызывается **в момент декорирования**, а не при вызове функции.

Это важное отличие от предыдущего примера.

---

### Почему класс здесь удобнее?

Потому что:

* параметры (`level`) хранятся в объекте
* структура становится линейной
* код проще расширять

Нет вложенных функций в три уровня.

---

На этих примерах видно, что `__call__` — это не просто синтаксическая возможность.

Он позволяет:

* объединить данные и поведение
* хранить состояние
* строить повторно используемые компоненты
* реализовывать паттерны проектирования

Фактически, это инструмент для создания **умных обёрток** над функциями.

---

## Когда использовать `__call__`

Используйте `__call__`, если:

* объект логически должен «вызываться»
* нужно хранить состояние между вызовами
* вы пишете декоратор (особенно сложный)
* вам нужна конфигурация (параметры)
* вы строите архитектурные компоненты (логирование, кеш, прокси)

Не используйте его, если:

* достаточно обычной функции
* нет состояния
* нет необходимости в расширении

Очень важно не использовать магические методы «просто потому что можно».

### **Задача 7**

Реализуйте класс `Cache`, который кеширует результаты функции декорируя ее.

Для сохранения кэша нужно использовать словарь `__cache`.

Если функция вызывается с одинаковыми аргументами больше одного раза, результат работы функции должен браться из словаря.

Если аргументами в функцию будут передаваться изменяемые типы данных (например список), программа не должна завершаться с ошибкой.

**Пример использования**:

```python
@Cache
def add(a, b):
    print("Считаем...")
    for i in range(5):
        time.sleep(1)
        print(f'{i + 1}')
    return a + b

print(add(2, 3))
print(add(2, 3))
```
