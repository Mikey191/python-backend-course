## Что такое веб-приложение

Когда вы открываете в браузере страницу или мобильное приложение получает данные из интернета, за этим стоит простая схема: 

> одна программа просит что-то у другой программы, и та отвечает.

Программа, которая просит, называется **клиентом**. 

Программа, которая отвечает, называется **сервером**. 

Именно **сервер** — это то, что пишет бэкенд-разработчик на Python.

```
Браузер / мобильное приложение       Сервер на Python
        (клиент)                         (бэкенд)

           ──── запрос (request) ────▶
           ◀─── ответ  (response) ────
```

Это взаимодействие происходит по протоколу HTTP. Детали протокола сейчас неважны — важно понять, что каждый цикл «запрос — ответ» является атомарным: клиент отправляет ровно один запрос, сервер возвращает ровно один ответ.

---

## Запрос (Request)

**Запрос** — это сообщение от клиента к серверу. Он содержит несколько обязательных частей.

**Метод** — это глагол, который описывает намерение клиента. Основные методы:

- `GET` — получить данные (открыть страницу, загрузить список товаров)
- `POST` — отправить данные (зарегистрировать пользователя, оформить заказ)
- `PUT` / `PATCH` — обновить существующие данные
- `DELETE` — удалить данные

**Путь (path)** — это адрес конкретного ресурса на сервере. Например, `/api/users/42` означает «пользователь с идентификатором 42».

**Заголовки (headers)** — это метаданные запроса: служебная информация о том, кто делает запрос, в каком формате ожидается ответ, есть ли токен авторизации. Заголовки — это словарь: ключ и значение.

**Тело (body)** — данные, которые клиент передаёт серверу. Присутствует только в запросах типа `POST`, `PUT`, `PATCH`. Обычно передаётся в формате JSON, который в Python представляется обычным словарём.

В коде запрос часто моделируется как объект:

```python
# Так выглядит объект запроса в упрощённом виде
request = {
    "method": "POST",
    "path": "/api/users",
    "headers": {
        "Content-Type": "application/json",
        "Authorization": "Bearer eyJhbGciOiJIUzI1NiJ9..."
    },
    "body": {
        "username": "alice",
        "email": "alice@example.com",
        "password": "secret123"
    }
}
```

Во фреймворках Django, Flask и FastAPI запрос — это не словарь, а объект класса с атрибутами: `request.method`, `request.path`, `request.headers`, `request.body`. 

Именно поэтому в примерах этого модуля мы будем создавать классы с именами вроде `HTTPRequest` или `Request` — это упрощённые модели того, с чем вы будете работать в реальных фреймворках.

## Ответ (Response)

Ответ — это сообщение от сервера к клиенту. Его основные части:

**Статусный код** — трёхзначное число, которое сообщает клиенту, как завершилась обработка запроса. Это не произвольное число — это стандарт, и клиент (браузер, мобильное приложение) принимает решения на основании этого кода.

Основные группы статусных кодов:

| Диапазон | Смысл | Примеры |
|---|---|---|
| 2xx | Успех | 200 OK, 201 Created, 204 No Content |
| 3xx | Перенаправление | 301 Moved Permanently, 302 Found |
| 4xx | Ошибка клиента | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found |
| 5xx | Ошибка сервера | 500 Internal Server Error, 503 Service Unavailable |

Именно поэтому в примерах модуля вы встретите конструкции вида `if response.status_code == 200:` или `if 200 <= status_code < 300:` — код проверяет, успешно ли обработан запрос.

**Тело ответа (body)** — данные, которые сервер возвращает клиенту. Как правило, это JSON: список объектов, один объект или сообщение об ошибке. В Python это словарь или список словарей.

**Заголовки ответа** — метаданные: тип содержимого, параметры кеширования, служебная информация.

```python
# Так выглядит объект ответа в упрощённом виде
response = {
    "status_code": 200,
    "headers": {
        "Content-Type": "application/json"
    },
    "body": {
        "id": 42,
        "username": "alice",
        "email": "alice@example.com"
    }
}
```

## Почему данные передаются в виде словарей

В примерах этого модуля вы увидите, что данные часто передаются и хранятся в виде словарей Python. Причина простая: формат JSON, который является стандартом передачи данных в вебе, при загрузке в Python превращается именно в словарь.

```python
import json

# Данные пришли с клиента в формате JSON (строка):
raw_data = '{"username": "alice", "age": 28, "is_active": true}'

# После обработки это становится словарём Python:
data = json.loads(raw_data)
print(data)
# {'username': 'alice', 'age': 28, 'is_active': True}
print(type(data))   # <class 'dict'>
```

Когда сервер возвращает данные клиенту — происходит обратное: словарь Python превращается в строку JSON. Именно поэтому словари — это основной способ передачи структурированных данных между клиентом и сервером, и именно поэтому аргументы в примерах часто выглядят как словари или именованные параметры.

## Кто такой User и зачем он нужен

Почти в каждом веб-приложении есть пользователи — люди или системы, которые взаимодействуют с сервером. Объект пользователя обычно содержит:

- идентификатор (`id`) — уникальный номер в базе данных
- имя или логин (`username`, `name`)
- адрес электронной почты (`email`)
- роль (`role`) — что этому пользователю разрешено делать: `"admin"`, `"moderator"`, `"user"`
- флаг активности (`is_active`) — заблокирован ли аккаунт

В реальном приложении объект пользователя создаётся из базы данных и прикрепляется к каждому входящему запросу: сервер «узнаёт», кто именно делает запрос, и принимает решение — разрешить действие или отказать.

Именно поэтому в примерах этого модуля классы `User`, `UserRecord` и подобные будут появляться как аргументы функций и как атрибуты запроса. Они моделируют реальные объекты, которые живут в любом бэкенд-приложении.

## Что такое конфигурация приложения

Веб-приложение не запускается с жёстко прописанными параметрами вроде адреса базы данных или секретного ключа — это сделало бы его негибким и небезопасным. Вместо этого параметры хранятся отдельно и загружаются при запуске.

Конфигурация — это набор настроек, которые определяют поведение приложения в конкретной среде. Например:

```python
config = {
    "DEBUG": False,
    "DATABASE_URL": "postgresql://localhost:5432/myapp",
    "SECRET_KEY": "a-very-long-random-string",
    "ALLOWED_HOSTS": ["example.com", "www.example.com"],
    "EMAIL_HOST": "smtp.gmail.com",
}
```

В реальных проектах конфигурация загружается из переменных окружения (файл `.env`) или из специального файла настроек. 

Объект конфигурации создаётся один раз при запуске приложения и используется во всех модулях. Именно поэтому конфигурация — один из классических примеров паттерна Singleton, который разбирался в начале курса.

## Что такое соединение с базой данных

Данные веб-приложения хранятся не в памяти программы, а в базе данных — отдельной системе, которая работает независимо от сервера. Для того чтобы прочитать или записать данные, сервер должен установить соединение с базой данных.

Соединение — это ресурс, у которого есть состояние: оно может быть `открытым`, `закрытым` или `разорванным` из-за ошибки. 

Работа с соединением всегда выглядит одинаково:

1. Открыть соединение.
2. Выполнить запрос.
3. Закрыть соединение — обязательно, даже если на шаге 2 возникла ошибка.

Именно это поведение «обязательного закрытия» моделируется через контекстные менеджеры (`with`-блоки), которые разбираются в уроке 15. 

Объект `DatabaseConnection` в примерах этого модуля — это упрощённая модель того, как работают реальные библиотеки для подключения к PostgreSQL, MySQL или SQLite.

## Краткий словарь терминов

Ниже — минимальный набор понятий, которые встречаются в примерах модуля 3. Сохраните эту страницу как справочник.

**Request (запрос)** — сообщение от клиента к серверу. Содержит метод (GET, POST...), путь, заголовки и, возможно, тело с данными.

**Response (ответ)** — сообщение от сервера к клиенту. Содержит статусный код, заголовки и тело с данными или описанием ошибки.

**Статусный код** — трёхзначное число в ответе. 2xx = успех, 4xx = ошибка клиента, 5xx = ошибка сервера.

**Headers (заголовки)** — словарь метаданных в запросе или ответе. Содержит служебную информацию: тип данных, токен авторизации, параметры кеширования.

**Body (тело)** — основные данные запроса или ответа. В Python — словарь, загруженный из JSON.

**JSON** — текстовый формат передачи данных. При загрузке в Python превращается в словарь или список.

**Endpoint** — конкретный адрес (путь) на сервере, обрабатывающий запросы определённого типа. Например, `POST /api/users` — это endpoint для создания пользователя.

**Токен авторизации** — строка, которую клиент передаёт в заголовке запроса, чтобы сервер понял, кто делает запрос. Аналог пропуска.

**Middleware** — промежуточный обработчик, который выполняется для каждого запроса до передачи его основному обработчику. Используется для логирования, проверки авторизации, сжатия данных.

**ORM (Object-Relational Mapping)** — библиотека, которая позволяет работать с базой данных через объекты Python, не писая SQL вручную. Django ORM — пример такой библиотеки.

## Как это связано с ООП и магическими методами

Вся перечисленная выше инфраструктура — запросы, ответы, конфигурации, соединения — реализована в реальных фреймворках через классы. И каждый из этих классов использует магические методы, которые мы будем изучать в модуле 3.

Объект запроса использует `__getattr__`, чтобы обращение к заголовку выглядело как `request.content_type`, а не `request.headers["content-type"]`. Объект ответа реализует `__bool__`, чтобы `if response:` означало «запрос прошёл успешно». Объект соединения с базой данных реализует `__enter__` и `__exit__`, чтобы работать внутри блока `with` и гарантировать закрытие соединения. Объект QuerySet в Django реализует `__iter__`, чтобы его можно было обойти в цикле `for`.

Именно поэтому примеры в уроках специально построены на этих объектах: они не случайные абстракции, а конкретные классы, прообразы которых вы напишете сами уже в следующих курсах.
