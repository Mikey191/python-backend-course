# Урок 9. Целостность данных и типы

## Что такое целостность данных

В Уроке 8 мы научились изменять данные. И сразу увидели что данные легко испортить — `UPDATE` без `WHERE`, вставка дубликата, удаление не той строки. Всё это **нарушения целостности данных**.

**Целостность данных** — это гарантия того что данные в базе корректны, согласованы и соответствуют правилам предметной области. Например:
- У каждого заказа должен быть пользователь
- Email не может повторяться
- Цена товара не может быть отрицательной
- Количество товара не может быть `NULL`

Целостность обеспечивается на двух уровнях: **ограничениями** (constraints) — правилами на уровне схемы, и **транзакциями** — механизмом атомарного выполнения операций. Разберём оба.

---

## Ограничения (Constraints)

Ограничения объявляются при создании таблицы и автоматически проверяются СУБД при каждой операции `INSERT` и `UPDATE`. Если данные нарушают ограничение — операция отклоняется с ошибкой. Это первая линия защиты данных.

В Уроке 2 мы кратко познакомились с `NOT NULL` и `PRIMARY KEY`. Сейчас разберём полный набор ограничений и то как они работают вместе.

### NOT NULL

Гарантирует что столбец не может содержать `NULL` — значение обязательно должно быть указано при вставке:

```sql
CREATE TABLE products (
    id    INTEGER PRIMARY KEY AUTOINCREMENT,
    name  TEXT    NOT NULL,   -- название обязательно
    price REAL    NOT NULL,   -- цена обязательна
    stock INTEGER             -- остаток необязателен (может быть NULL)
);
```

Попытка вставить `NULL` в защищённый столбец:

```sql
INSERT INTO products (name, price) VALUES (NULL, 1000.00);
-- Ошибка: NOT NULL constraint failed: products.name
```

Когда использовать `NOT NULL`: для всех данных без которых запись теряет смысл. Товар без названия или без цены — это не товар. Пользователь без email — непонятная сущность.

### UNIQUE

Гарантирует что значение столбца уникально во всей таблице. Два пользователя не могут иметь одинаковый email:

```sql
CREATE TABLE users (
    id    INTEGER PRIMARY KEY AUTOINCREMENT,
    email TEXT    NOT NULL UNIQUE  -- не NULL и уникален
);
```

```sql
INSERT INTO users (email) VALUES ('alice@mail.ru');
INSERT INTO users (email) VALUES ('alice@mail.ru');
-- Ошибка: UNIQUE constraint failed: users.email
```

`UNIQUE` допускает несколько строк с `NULL` — `NULL` не равен `NULL`, поэтому два `NULL` не считаются дубликатами. Именно поэтому в нашей базе `email` объявлен `NOT NULL UNIQUE` — оба ограничения вместе дают полную гарантию.

**Составной UNIQUE** — уникальность комбинации столбцов. Например, один товар может быть в одном заказе только один раз:

```sql
CREATE TABLE order_items (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    order_id   INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    quantity   INTEGER NOT NULL DEFAULT 1,
    UNIQUE (order_id, product_id)  -- комбинация уникальна
);
```

Это означает что в одном заказе один и тот же товар не может появиться дважды — вместо этого нужно увеличить `quantity`.

### DEFAULT

Задаёт значение по умолчанию — если при вставке столбец не указан, он получит это значение:

```sql
CREATE TABLE orders (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id    INTEGER NOT NULL,
    status     TEXT    NOT NULL DEFAULT 'pending',
    created_at TEXT    NOT NULL DEFAULT (date('now'))
);
```

```sql
-- Не указываем status и created_at
INSERT INTO orders (user_id) VALUES (1);

SELECT id, user_id, status, created_at FROM orders ORDER BY id DESC LIMIT 1;
```

```
┌────┬─────────┬─────────┬────────────┐
│ id │ user_id │ status  │ created_at │
├────┼─────────┼─────────┼────────────┤
│ 19 │       1 │ pending │ 2024-06-25 │
└────┴─────────┴─────────┴────────────┘
```

`DEFAULT (date('now'))` — это встроенная функция SQLite которая возвращает текущую дату. `DEFAULT` может принимать как статическое значение (`'pending'`, `0`) так и выражение в скобках.

В нашем `seed.sql` `stock` объявлен с `DEFAULT 0` — поэтому в Уроке 8 новые товары без указания `stock` получали ноль автоматически.

### CHECK

Позволяет задать произвольное условие которому должно соответствовать значение. Это самое гибкое ограничение:

```sql
CREATE TABLE products (
    id       INTEGER PRIMARY KEY AUTOINCREMENT,
    name     TEXT    NOT NULL,
    price    REAL    NOT NULL CHECK (price > 0),       -- цена только положительная
    stock    INTEGER NOT NULL DEFAULT 0 CHECK (stock >= 0),  -- остаток не отрицательный
    discount REAL    DEFAULT 0 CHECK (discount >= 0 AND discount <= 100)  -- скидка 0-100%
);
```

```sql
INSERT INTO products (name, price, stock) VALUES ('Тест', -500.00, 10);
-- Ошибка: CHECK constraint failed: price > 0
```

```sql
INSERT INTO products (name, price, stock) VALUES ('Тест', 500.00, -1);
-- Ошибка: CHECK constraint failed: stock >= 0
```

`CHECK` проверяется при каждом `INSERT` и `UPDATE`. Это хорошее место для бизнес-правил которые должны соблюдаться всегда — вне зависимости от того через какой код идёт запрос в базу.

### Именованные ограничения

Ограничениям можно давать имена через `CONSTRAINT`. Это удобно когда нужно понять из сообщения об ошибке что именно нарушено:

```sql
CREATE TABLE products (
    id    INTEGER PRIMARY KEY AUTOINCREMENT,
    name  TEXT    NOT NULL,
    price REAL    NOT NULL,
    stock INTEGER NOT NULL DEFAULT 0,
    CONSTRAINT price_positive CHECK (price > 0),
    CONSTRAINT stock_non_negative CHECK (stock >= 0)
);
```

```sql
INSERT INTO products (name, price, stock) VALUES ('Тест', -100, 5);
-- Ошибка: CHECK constraint failed: price_positive
```

Сообщение теперь содержит имя ограничения — сразу понятно что именно нарушено.

### Все ограничения вместе — пример схемы

Так выглядит таблица с полным набором ограничений:

```sql
CREATE TABLE products (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT    NOT NULL,
    price       REAL    NOT NULL,
    stock       INTEGER NOT NULL DEFAULT 0,
    category_id INTEGER NOT NULL,
    FOREIGN KEY (category_id) REFERENCES categories(id),
    CHECK (price > 0),
    CHECK (stock >= 0)
);
```

Каждое ограничение отвечает за свою часть целостности:
- `PRIMARY KEY` — уникальность и идентификация строки
- `NOT NULL` — обязательность данных
- `DEFAULT` — разумные значения при вставке
- `FOREIGN KEY` — ссылочная целостность
- `CHECK` — бизнес-правила

## Транзакции

Ограничения защищают каждую отдельную операцию. Но что если нужно выполнить несколько операций которые должны завершиться **все вместе или не выполниться вовсе**?

Классический пример — перевод денег между счетами:

```sql
-- Снять 1000 руб со счёта А
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
-- Зачислить 1000 руб на счёт Б
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
```

Если после первого `UPDATE` произойдёт сбой — деньги списаны, но не зачислены. Данные несогласованны. Транзакция решает эту проблему.

### BEGIN, COMMIT, ROLLBACK

**Транзакция** — это группа операций которая выполняется как единое целое:

```sql
BEGIN;
    -- первая операция
    -- вторая операция
    -- третья операция
COMMIT;  -- зафиксировать все изменения
```

Если что-то пошло не так:

```sql
BEGIN;
    -- первая операция (выполнена)
    -- вторая операция (ошибка!)
ROLLBACK;  -- отменить все изменения начиная с BEGIN
```

### Пример с нашей базой

Добавление заказа — это всегда минимум две операции: создать заказ, добавить позиции. Они должны выполниться вместе:

```sql
BEGIN;

INSERT INTO orders (user_id, status, created_at)
VALUES (1, 'pending', '2024-06-25');

INSERT INTO order_items (order_id, product_id, quantity, price_at_time)
VALUES (
    (SELECT MAX(id) FROM orders),
    1, 1, 75000.00
);

INSERT INTO order_items (order_id, product_id, quantity, price_at_time)
VALUES (
    (SELECT MAX(id) FROM orders),
    7, 1, 5500.00
);

COMMIT;
```

Если любая из трёх операций завершится ошибкой — `ROLLBACK` отменит все предыдущие. База данных вернётся в состояние до `BEGIN`.

Что бы правильно запустить транзакцию в SQLiteStudio вы должны запускать SQL как **скрипт целиком**, а не отдельный statement. Нужно **выделить весь SQL целиком** и нажать горячую клавишу **F9**.

### Транзакция с явным ROLLBACK

```sql
BEGIN;

UPDATE products SET price = 0 WHERE category_id = 1;

-- Проверяем что получилось
SELECT name, price FROM products WHERE category_id = 1;
-- Видим что цены стали нулевыми — это не то что нам нужно

ROLLBACK;  -- отменяем изменения

-- Проверяем — цены вернулись
SELECT name, price FROM products WHERE category_id = 1;
```

`ROLLBACK` позволяет "попробовать" изменения и отменить их если результат не устраивает. Это полезно при экспериментах в SQLiteStudio.

Важно понимать: `ROLLBACK` отменяет только незавершённую транзакцию — то есть изменения после последнего `BEGIN`. Если выполнить `COMMIT`, изменения фиксируются окончательно и откатить их через `ROLLBACK` уже нельзя.

```sql
BEGIN;

UPDATE products SET price = 0;

COMMIT;   -- изменения сохранены

ROLLBACK; -- уже ничего не отменит
```

### Автоматические транзакции

SQLite автоматически оборачивает каждую операцию в транзакцию если вы не указали `BEGIN` явно. Поэтому одиночный `INSERT` или `UPDATE` — это всегда атомарная операция: либо выполнилась полностью, либо нет.

Явный `BEGIN` нужен только когда несколько операций должны быть атомарными вместе.

## Типизация в SQLite — почему она мягкая

В Уроке 2 мы познакомились с четырьмя базовыми типами SQLite: `INTEGER`, `REAL`, `TEXT`, `BLOB`. Пришло время разобраться почему SQLite обращается с типами иначе чем большинство СУБД — и почему это важно понимать.

### Система типов SQLite: affinity

SQLite использует концепцию **type affinity** — "сродство к типу". Это означает что тип столбца — это не жёсткое ограничение, а предпочтение. SQLite попытается привести значение к нужному типу, но если не получится — сохранит как есть.

```sql
CREATE TABLE test (col INTEGER);

INSERT INTO test VALUES (42);       -- OK: целое число
INSERT INTO test VALUES (3.14);     -- OK: SQLite сохранит как REAL
INSERT INTO test VALUES ('hello');  -- OK: SQLite сохранит как TEXT
INSERT INTO test VALUES ('42');     -- OK: SQLite преобразует в INTEGER 42
```

Это не ошибка — это намеренное поведение SQLite. Вот что происходит с каждым случаем:

| Значение | Тип столбца | Что сохранится |
|---|---|---|
| `42` | `INTEGER` | `42` (INTEGER) |
| `3.14` | `INTEGER` | `3.14` (REAL — не вмещается в INTEGER) |
| `'hello'` | `INTEGER` | `'hello'` (TEXT — не конвертируется) |
| `'42'` | `INTEGER` | `42` (TEXT конвертируется в INTEGER) |

### Пять классов хранения

Реально SQLite хранит данные в пяти классах — независимо от объявленного типа столбца:

| Класс хранения | Что хранит |
|---|---|
| `NULL` | Отсутствие значения |
| `INTEGER` | Целое число (1, 2, 3, 4, 6 или 8 байт) |
| `REAL` | Число с плавающей точкой (8 байт, IEEE 754) |
| `TEXT` | Строка в кодировке UTF-8 |
| `BLOB` | Бинарные данные — хранятся без изменений |

Объявленный тип столбца (`INTEGER`, `TEXT`, `REAL`) определяет только правило преобразования при вставке — не класс хранения реального значения.

### Почему это важно на практике

Мягкая типизация SQLite — это удобство при разработке и обучении. Но она скрывает ошибки которые в строгих СУБД были бы сразу видны.

SQLite не имеет отдельного типа данных для дат и времени. Обычно даты хранятся как `TEXT` в формате `YYYY-MM-DD`. Этот формат выбран не случайно: строки в таком виде лексикографически сортируются так же как даты по времени, поэтому операторы сравнения (`>`, `<`) и `ORDER BY` работают корректно.

```sql
-- В нашей базе даты хранятся как TEXT
-- SQLite не знает что это дата — для него это просто строка
SELECT * FROM orders WHERE created_at > '2024-03-01';
-- Работает корректно только потому что формат YYYY-MM-DD
-- лексикографически сравним с хронологическим порядком
```

```sql
-- Если бы кто-то вставил дату в другом формате:
INSERT INTO orders (user_id, status, created_at)
VALUES (1, 'pending', '25.06.2024');  -- SQLite примет без ошибки
-- Но сравнение с '2024-06-01' даст неверный результат
```

В PostgreSQL ситуация другая: там существуют отдельные типы `DATE`, `TIMESTAMP` и другие. База данных понимает что значение является датой и проверяет корректность формата. Строка `'25.06.2024'` без явного преобразования вызвала бы ошибку ещё на этапе INSERT.

### Приведение типов через CAST

Для явного преобразования значения к нужному типу используется оператор `CAST`:

```sql
SELECT CAST('123' AS INTEGER);
-- Результат: 123

SELECT CAST(10.8 AS INTEGER);
-- Результат: 10
```

`CAST` позволяет временно привести значение к нужному типу прямо внутри запроса.

Это особенно полезно в `CHECK`-ограничениях. Например, SQLite может сохранить дробное число `1.5` в столбец `INTEGER`. Чтобы запретить это, можно явно проверить что значение после приведения к `INTEGER` не изменилось:

```sql
quantity INTEGER NOT NULL
CHECK (
    quantity > 0
    AND quantity = CAST(quantity AS INTEGER)
)
```

Логика проверки:

* `CAST(quantity AS INTEGER)` отбрасывает дробную часть
* если число было целым (`5`) — результат останется `5`
* если число было дробным (`1.5`) — результат станет `1`
* сравнение вернёт `FALSE`, и SQLite отклонит вставку

```sql
CAST(5 AS INTEGER)     = 5     -- TRUE
CAST(1.5 AS INTEGER)   = 1     -- FALSE при сравнении с 1.5
```

### Практическое правило

Мягкость SQLite не означает что типами можно пренебрегать. Правильная привычка:
- Всегда объявляйте типы столбцов осмысленно
- Следите за тем что вставляете — СУБД не всегда вас остановит
- Помните что при переходе на PostgreSQL типизация станет строгой

Это и есть тот "спойлер к PostgreSQL" — там за типами следит сама СУБД и ошибки будут видны раньше.
