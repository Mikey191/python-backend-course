# Урок 7. Подзапросы и CTE

## Что такое подзапрос

Подзапрос — это запрос внутри другого запроса. Результат внутреннего запроса используется внешним запросом как данные для фильтрации, вычисления или источник строк.

```sql
-- Внешний запрос использует результат внутреннего
SELECT name, price FROM products
WHERE price > (SELECT AVG(price) FROM products);
```

Здесь `(SELECT AVG(price) FROM products)` — подзапрос. Он выполняется первым и возвращает одно число — среднюю цену. Затем внешний запрос использует это число как условие фильтрации.

```
┌──────────────────────────────┬──────────┐
│ name                         │ price    │
├──────────────────────────────┼──────────┤
│ Ноутбук Lenovo IdeaPad       │ 75000.00 │
│ Ноутбук ASUS VivoBook        │ 68000.00 │
│ Планшет Apple iPad           │ 55000.00 │
│ Смартфон Samsung Galaxy      │ 45000.00 │
│ Кресло офисное Hara          │ 32000.00 │
│ Монитор LG 27"               │ 32000.00 │
│ Наушники Sony WH-1000XM5     │ 28000.00 │
│ Смартфон Xiaomi Redmi        │ 22000.00 │
└──────────────────────────────┴──────────┘
```

Подзапросы бывают трёх видов в зависимости от того где они стоят в запросе: в `WHERE`, в `FROM` и в `SELECT`. Разберём каждый.

## Подзапрос в WHERE

Самое частое применение — использовать подзапрос как условие фильтрации.

### Скалярный подзапрос — одно значение

Когда подзапрос возвращает ровно одно значение, его можно использовать с обычными операторами сравнения (`=`, `>`, `<`, `!=`):

```sql
-- Товары дороже среднего
SELECT name, price FROM products
WHERE price > (SELECT AVG(price) FROM products)
ORDER BY price DESC;
```

```sql
-- Самый последний заказ
SELECT id, user_id, status FROM orders
WHERE created_at = (SELECT MAX(created_at) FROM orders);
```

```
┌────┬─────────┬─────────┐
│ id │ user_id │ status  │
├────┼─────────┼─────────┤
│ 16 │      13 │ pending │
└────┴─────────┴─────────┘
```

### Подзапрос с IN — список значений

Когда подзапрос возвращает несколько строк, его используют с `IN`. В Уроке 3 мы писали в `IN` список значений вручную — теперь этот список генерирует запрос:

```sql
-- Пользователи у которых есть хотя бы один доставленный заказ
SELECT name, email FROM users
WHERE id IN (
    SELECT user_id FROM orders
    WHERE status = 'delivered'
);
```

```
┌──────────────────┬──────────────────────┐
│ name             │ email                │
├──────────────────┼──────────────────────┤
│ Алексей Смирнов  │ alexey@gmail.com     │
│ Мария Иванова    │ maria@yandex.ru      │
│ Дмитрий Козлов   │ dmitry@mail.ru       │
│ Елена Новикова   │ elena@gmail.com      │
│ ...              │ ...                  │
└──────────────────┴──────────────────────┘
```

Внутренний запрос `SELECT user_id FROM orders WHERE status = 'delivered'` возвращает список `user_id`. Внешний запрос использует этот список как фильтр.

`NOT IN` — пользователи у которых нет ни одного доставленного заказа:

```sql
SELECT name, email FROM users
WHERE id NOT IN (
    SELECT user_id FROM orders
    WHERE status = 'delivered'
);
```

### Подзапрос с EXISTS

`EXISTS` проверяет не значения, а сам факт существования строк в подзапросе. Возвращает `TRUE` если подзапрос вернул хотя бы одну строку, и `FALSE` если пустой результат.

Особенность `EXISTS` - очень часто используется с **коррелированными подзапросами**: он ссылается на столбец из внешнего запроса через псевдоним. Для каждой строки внешнего запроса внутренний выполняется заново:

```sql
-- Пользователи у которых есть хотя бы один заказ
SELECT u.name FROM users AS u
WHERE EXISTS (
    SELECT 1 FROM orders AS o
    WHERE o.user_id = u.id  -- ссылка на строку внешнего запроса
);
```

`SELECT 1` внутри `EXISTS` — это стандартная идиома. Нас не интересует что именно возвращает подзапрос — только факт существования строк. Поэтому возвращаем просто `1`.

`NOT EXISTS` — пользователи без единого заказа:

```sql
SELECT u.name, u.email FROM users AS u
WHERE NOT EXISTS (
    SELECT 1 FROM orders AS o
    WHERE o.user_id = u.id
);
```
*Если в БД нет такого пользователя, то можно его добавить для проверки.*

Одна из ловушек SQL: сравнение с `NULL` через `IN`/`NOT IN` даёт `UNKNOWN` а не `TRUE` или `FALSE`. `NOT IN` с `NULL` в списке не вернёт ни одной строки. В такой ситуации безопаснее использовать `NOT EXISTS`.


### IN vs EXISTS — когда что использовать

Оба варианта часто дают одинаковый результат. Практическое правило:

| Ситуация | Что использовать |
|---|---|
| Подзапрос возвращает небольшой список значений | `IN` — читается проще |
| Нужно проверить сам факт существования | `EXISTS` — семантически точнее |
| Подзапрос может вернуть `NULL` в списке | `EXISTS` — `NOT IN` с `NULL` работает непредсказуемо |

## Подзапрос в FROM — вложенная таблица

Подзапрос можно поставить вместо имени таблицы в `FROM`. Результат подзапроса становится временной таблицей — с ней работает внешний запрос. Такой подзапрос называют **производной таблицей** (derived table) и он обязательно должен иметь псевдоним:

```sql
-- Средняя сумма заказа по каждому пользователю
-- Шаг 1 (подзапрос): посчитать сумму каждого заказа
-- Шаг 2 (внешний запрос): усреднить суммы по пользователю
SELECT
    user_id,
    ROUND(AVG(order_total), 2) AS avg_order_value
FROM (
    SELECT
        o.user_id,
        o.id AS order_id,
        SUM(oi.quantity * oi.price_at_time) AS order_total
    FROM orders AS o
    INNER JOIN order_items AS oi ON o.id = oi.order_id
    GROUP BY o.id, o.user_id
) AS order_totals
GROUP BY user_id
ORDER BY avg_order_value DESC;
```

```
┌─────────┬─────────────────┐
│ user_id │ avg_order_value │
├─────────┼─────────────────┤
│      10 │        83900.00 │
│       9 │        55000.00 │
│       1 │        42450.00 │
│ ...     │ ...             │
└─────────┴─────────────────┘
```

Подзапрос в `FROM` полезен когда нужно агрегировать уже агрегированные данные — SQLite не позволяет вложить одну агрегатную функцию в другую напрямую (`AVG(SUM(...))` — ошибка). Промежуточный результат выносится в подзапрос, а внешний запрос работает с ним как с обычной таблицей.

## Подзапрос в SELECT — вычисляемый столбец

Подзапрос можно использовать прямо в списке столбцов `SELECT`. Он выполняется для каждой строки результата и возвращает ровно одно значение:

```sql
-- Для каждого товара — его цена и средняя цена в его категории
SELECT
    p.name,
    p.price,
    (
        SELECT ROUND(AVG(p2.price), 2)
        FROM products AS p2
        WHERE p2.category_id = p.category_id
    ) AS category_avg_price
FROM products AS p
ORDER BY p.category_id, p.price DESC;
```

```
┌──────────────────────────────┬──────────┬────────────────────┐
│ name                         │ price    │ category_avg_price │
├──────────────────────────────┼──────────┼────────────────────┤
│ Ноутбук Lenovo IdeaPad       │ 75000.00 │           39166.67 │
│ Ноутбук ASUS VivoBook        │ 68000.00 │           39166.67 │
│ Планшет Apple iPad           │ 55000.00 │           39166.67 │
│ ...                          │ ...      │                ... │
└──────────────────────────────┴──────────┴────────────────────┘
```

Подзапрос в `SELECT` — тоже коррелированный: он ссылается на `p.category_id` из внешнего запроса. Используйте его с осторожностью на больших таблицах — он выполняется для каждой строки результата.

Подзапрос в `SELECT` выполняется для каждой строки результата и добавляет вычисляемый столбец. Он должен возвращать ровно одно значение. Подзапрос в `WHERE` выполняется один раз (или для каждой строки при `EXISTS`) и используется как условие фильтрации — возвращает одно значение или список.

## CTE — Common Table Expressions

Подзапросы работают, но когда их несколько или они вложены друг в друга — запрос становится трудно читаемым. **CTE** (Common Table Expression, обобщённое табличное выражение) решает эту проблему: позволяет дать подзапросу имя и вынести его в начало запроса.

Синтаксис:

```sql
WITH имя_cte AS (
    -- запрос
),
ещё_одно_cte AS (
    -- ещё запрос
)
SELECT ...
FROM имя_cte ...;
```

### Пример: тот же запрос через CTE

Перепишем подзапрос в `FROM` из предыдущего раздела:

```sql
-- С подзапросом — сложно читать
SELECT user_id, ROUND(AVG(order_total), 2) AS avg_order_value
FROM (
    SELECT o.user_id, o.id, SUM(oi.quantity * oi.price_at_time) AS order_total
    FROM orders AS o
    INNER JOIN order_items AS oi ON o.id = oi.order_id
    GROUP BY o.id, o.user_id
) AS order_totals
GROUP BY user_id
ORDER BY avg_order_value DESC;

-- С CTE — читается как текст
WITH order_totals AS (
    SELECT
        o.user_id,
        o.id AS order_id,
        SUM(oi.quantity * oi.price_at_time) AS order_total
    FROM orders AS o
    INNER JOIN order_items AS oi ON o.id = oi.order_id
    GROUP BY o.id, o.user_id
)
SELECT
    user_id,
    ROUND(AVG(order_total), 2) AS avg_order_value
FROM order_totals
GROUP BY user_id
ORDER BY avg_order_value DESC;
```

Результат одинаков. Но CTE-версия читается сверху вниз: сначала определяем `order_totals`, потом используем его.

### Несколько CTE

CTE можно определять несколько — через запятую. Каждый следующий CTE может ссылаться на предыдущие:

```sql
-- Топ-3 самых покупаемых товара и их категории
WITH
-- Шаг 1: посчитать сколько раз заказан каждый товар
product_orders AS (
    SELECT
        product_id,
        SUM(quantity) AS total_quantity
    FROM order_items
    GROUP BY product_id
),
-- Шаг 2: взять топ-3
top_products AS (
    SELECT product_id, total_quantity
    FROM product_orders
    ORDER BY total_quantity DESC
    LIMIT 3
)
-- Шаг 3: добавить названия товара и категории
SELECT
    p.name          AS product_name,
    c.name          AS category_name,
    tp.total_quantity
FROM top_products AS tp
INNER JOIN products    AS p ON tp.product_id  = p.id
INNER JOIN categories  AS c ON p.category_id  = c.id
ORDER BY tp.total_quantity DESC;
```

```
┌──────────────────────┬─────────────┬────────────────┐
│ product_name         │ category_name│ total_quantity │
├──────────────────────┼─────────────┼────────────────┤
│ Футболка хлопок M    │ Одежда      │              3 │
│ Чистый код. Мартин   │ Книги       │              5 │
│ ...                  │ ...         │            ... │
└──────────────────────┴─────────────┴────────────────┘
```

Запрос читается как пошаговое описание логики — каждый CTE это один понятный шаг.

CTE определяются последовательно и каждый следующий может ссылаться на предыдущие. 

Это позволяет строить логику пошагово: первый CTE готовит данные, второй их фильтрует или агрегирует, третий добавляет детали.

## Подзапрос vs CTE — сравнение подходов

Оба инструмента решают одни и те же задачи. Выбор между ними — вопрос читаемости:

| | Подзапрос | CTE |
|---|---|---|
| Где размещается | Внутри запроса | В начале, через `WITH` |
| Читаемость | Снижается при вложенности | Высокая — логика линейна |
| Повторное использование | Нельзя — нужно дублировать | Можно ссылаться несколько раз |
| Производительность | Идентична в большинстве случаев | Идентична в большинстве случаев |
| Когда предпочтительнее | Простое условие в `WHERE` | Сложная логика, несколько шагов |

Практическое правило: если подзапрос помещается в одну строку и логика понятна — оставляйте подзапрос. Если запрос становится трудно читаемым или одни и те же данные нужны в нескольких местах — выносите в CTE.

## Полный порядок клаузул

```sql
WITH cte_name AS (
    ...
)
SELECT [DISTINCT] столбцы
FROM таблица / подзапрос / cte
JOIN ...
WHERE ...
GROUP BY ...
HAVING ...
ORDER BY ...
LIMIT ... OFFSET ...;
```

`WITH` всегда идёт самым первым — до `SELECT`.
