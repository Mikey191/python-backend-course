# Урок 35: Обработка ошибок в Python. Исключения и конструкция `try/except`

# МОДУЛЬ 1. ПРИРОДА ОШИБОК В PYTHON

## Блок 1. Что такое ошибка в программе?

Когда программа запускается, Python выполняет инструкции последовательно.
Если в процессе выполнения возникает ситуация, с которой интерпретатор не может справиться, выполнение **прерывается**, и возникает **исключение** (exception).

Исключение — это объект, который сигнализирует о проблеме во время выполнения программы.

Важно понимать:

> Ошибка — это не «что-то сломалось».
> Ошибка — это механизм защиты программы от некорректного состояния.

---

## Блок 2. Два принципиально разных типа ошибок

В Python нас интересуют два больших класса ошибок:

- SyntaxError — синтаксическая ошибка

- Runtime error — ошибка выполнения

Разберём каждую.

---

### Синтаксическая ошибка (SyntaxError)

Это ошибка структуры кода.
Python даже не начинает выполнять программу — он не может её разобрать.

Пример:

```python
def greet()
    print("Hello")
```

Результат:

```
SyntaxError: expected ':'
```

Что произошло?

* Python прочитал код
* Обнаружил, что после `def greet()` отсутствует `:`
* Остановил выполнение ДО запуска программы

⚠️ Программа вообще не стартовала.

**Синтаксические ошибки возникают**:

* при неправильном отступе
* при пропущенной скобке
* при отсутствии двоеточия
* при неправильной структуре выражения

---

### Ошибка выполнения (Runtime Error)

Это ошибка, которая возникает **во время работы программы**.

Программа уже начала выполняться.
Но в процессе произошло нечто недопустимое.

Пример:

```python
number = int("abc")
```

Результат:

```
ValueError: invalid literal for int() with base 10: 'abc'
```

Здесь:

* Код синтаксически корректен
* Программа стартовала
* Но во время выполнения возникла проблема

Это и есть `runtime error`.

---

### Самые распространённые runtime-ошибки

* `ValueError` — неправильное значение
* `TypeError` — несовместимые типы
* `ZeroDivisionError` — деление на ноль
* `IndexError` — выход за границы списка
* `KeyError` — отсутствующий ключ в словаре

---

## Блок 3. Что такое `traceback`?

Когда возникает `runtime-ошибка`, Python показывает не просто сообщение.

Он выводит **traceback**.

`Traceback` — это стек вызовов функций, который показывает путь программы до места ошибки.

Пример:

```python
def convert(text):
    return int(text)

def main():
    value = convert("abc")
    print(value)

main()
```

Результат:

```
Traceback (most recent call last):
  File "example.py", line 9, in <module>
    main()
  File "example.py", line 6, in main
    value = convert("abc")
  File "example.py", line 2, in convert
    return int(text)
ValueError: invalid literal for int() with base 10: 'abc'
```

---

### Как читать traceback (это критически важно)

Traceback читается **снизу вверх**.

Схема:

```
1. Тип ошибки (самая нижняя строка)
2. Строка, где ошибка произошла
3. Кто вызвал эту функцию
4. Кто вызвал того, кто вызвал функцию
...
```

Разберём:

```
ValueError: invalid literal for int() with base 10: 'abc'
```

Это тип и причина ошибки.

Выше:

```
File "example.py", line 2, in convert
    return int(text)
```

Ошибка произошла в функции `convert`, строка 2.

Ещё выше:

```
File "example.py", line 6, in main
    value = convert("abc")
```

Функция `main` вызвала `convert`.

Ещё выше:

```
File "example.py", line 9, in <module>
    main()
```

Программа стартовала и вызвала `main()`.

---

### Схема стека вызовов

```
main()  
   ↓
convert("abc")  
   ↓
int("abc")  ← ошибка
```

# МОДУЛЬ 2. КОНСТРУКЦИЯ `try/except`

Конструкция `try/except` - блок который позволяет **контролировать поведение программы при возникновении исключений**.

Цель этого модуля — научиться:

* перехватывать ошибки,
* разделять их по смыслу,
* управлять реакцией программы,
* не скрывать важные проблемы.

---

## Блок 4. Базовая конструкция `try/except`

### Зачем нужна эта конструкция?

По умолчанию, если в программе возникает ошибка выполнения (`runtime error`), Python:

* Прерывает выполнение.
* Показывает traceback.
* Завершает программу.

Иногда это поведение оправдано.

Но часто мы хотим:

* вывести понятное сообщение пользователю,
* продолжить работу программы,
* обработать ошибку и перейти к следующему действию.

Для этого используется конструкция:

```python
try:
    # код, в котором может возникнуть ошибка
except:
    # код, который выполнится при возникновении ошибки
```

---

### Простейший пример

```python
try:
    number = int(input("Введите число: "))
    print("Вы ввели:", number)
except:
    print("Произошла ошибка.")
```

Что происходит:

* Python пытается выполнить код в блоке `try`.
* Если ошибки нет — блок `except` пропускается.
* Если ошибка возникла — выполнение переходит в `except`.

---

**Важное замечание**

Конструкция `except:` без указания типа ошибки **перехватывает все исключения**.

Это означает:

* ValueError
* TypeError
* ZeroDivisionError
* FileNotFoundError
* и любые другие

Программа не упадёт.
Но вы можете потерять важную информацию.

---

## Блок 5. Обработка конкретных ошибок

Рассмотрим пример:

```python
try:
    number = int(input("Введите число: "))
    result = 10 / number
    print("Результат:", result)
except:
    print("Произошла ошибка.")
```

Здесь могут возникнуть:

* `ValueError` — если введён нечисловой текст
* `ZeroDivisionError` — если введён 0

> Но пользователь увидит одно и то же сообщение. Это плохая практика.

---

### Почему `except:` — плохая идея?

1. Скрывает тип ошибки.
2. Усложняет отладку.
3. Делает программу непредсказуемой.
4. Может скрыть критические ошибки.

---

### Правильный подход

Указывать **конкретный тип исключения**:

```python
try:
    number = int(input("Введите число: "))
    result = 10 / number
    print("Результат:", result)

except ValueError:
    print("Ошибка: нужно ввести число.")

except ZeroDivisionError:
    print("Ошибка: деление на ноль невозможно.")
```

Теперь:

* Каждая ошибка обрабатывается отдельно.
* Логика понятна.
* Поведение программы предсказуемо.

---

## Блок 6. Несколько ошибок

Рассмотрим два распространённых сценария.

---

### Сценарий 1. Объединение нескольких ошибок в одном `except`

Иногда по смыслу разные ошибки обрабатываются одинаково.

Пример:

```python
try:
    value = int(input("Введите индекс: "))
    items = [10, 20, 30]
    print(items[value])

except (ValueError, IndexError):
    print("Некорректный ввод данных.")
```

Здесь:

* `ValueError` — если введено не число
* `IndexError` — если индекс вне диапазона

> С точки зрения пользователя — это одна логическая проблема: "Некорректные данные". Поэтому объединение оправдано.

---

### Сценарий 2. Разделение по разным блокам

Если ошибки имеют разный смысл — их нужно разделить:

```python
try:
    value = int(input("Введите индекс: "))
    items = [10, 20, 30]
    print(items[value])

except ValueError:
    print("Ошибка: введите целое число.")

except IndexError:
    print("Ошибка: индекс вне диапазона.")
```

Теперь сообщения разные.

---

### Остаточный блок (перехват всех остальных ошибок)

Иногда мы хотим:

* Обработать важные ошибки отдельно.
* Все остальные — в общем блоке.

Пример:

```python
try:
    value = int(input("Введите число: "))
    result = 100 / value
    print(result)

except ValueError:
    print("Введите корректное число.")

except ZeroDivisionError:
    print("На ноль делить нельзя.")

except Exception:
    print("Произошла непредвиденная ошибка.")
```

`Exception` — базовый класс большинства ошибок выполнения.

Этот блок сработает для любых других `runtime-исключений`.

---

### Важно помнить порядок

Блоки `except` проверяются сверху вниз. Поэтому он должен идти последним.

Если сначала написать:

```python
except Exception:
```

то более конкретные блоки ниже никогда не выполнятся.

---

# МОДУЛЬ 3. ТЕКСТ ОШИБКИ И ОПЕРАТОР `raise`

**Переход от обработки аварий к управлению логикой программы**

Следующий шаг — научиться **управлять ошибками**.

До этого момента мы:

> ловили ошибки, которые происходили сами.

Теперь мы будем:

> создавать ошибки осознанно, чтобы управлять логикой программы.

---

## Блок 8. `except Exception as e`

Когда мы пишем:

```python
except ValueError:
    print("Ошибка ввода")
```

мы просто знаем, **что ошибка произошла**.

Но Python при возникновении исключения создаёт **объект ошибки**.

Этот объект **содержит**:

* тип ошибки
* текст ошибки
* дополнительную информацию

**Мы можем получить этот объект**:

```python
except Exception as e:
    print(e)
```

### Что такое `e`?

`e` — это **экземпляр класса исключения**. 

Фактически происходит следующее:

```
Произошла ошибка
        ↓
Python создал объект ошибки
        ↓
Мы сохранили его в переменную e
        ↓
Можем использовать его текст
```

**Рассмотрим код**:

```python
try:
    number = int(input("Введите число: "))
    result = 10 / number
except Exception as e:
    print("Произошла ошибка:", e)
```

Если пользователь введёт `abc`, мы получим:

```
Произошла ошибка: invalid literal for int() with base 10: 'abc'
```

Здесь `e` содержит технический текст ошибки.

---

### Зачем это нужно?

Иногда мы хотим:

* показать пользователю **простое сообщение**
* но сохранить реальный текст ошибки для анализа

### Разделение сообщений

```
Пользователь → получает понятное сообщение
Система      → получает технический текст ошибки
```

Пример:

```python
import logging

logging.basicConfig(filename="app.log", level=logging.ERROR)

try:
    number = int(input("Введите число: "))
    result = 10 / number
except Exception as e:
    print("Ошибка ввода данных.")
    logging.error(f"Техническая ошибка: {e}")
```

Теперь:

* пользователь видит аккуратное сообщение
* в файл `app.log` записывается реальный текст ошибки

Это уже уровень профессионального кода.

---

## Блок 9. Оператор `raise`

До этого момента исключения происходили сами.

Теперь мы научимся их **создавать вручную**.

### Что делает `raise`?

`raise` генерирует исключение.

```python
raise ValueError("Некорректное значение")
```

Происходит:

```
Выполнение программы
        ↓
Мы вызываем raise
        ↓
Создаётся объект ошибки
        ↓
Программа ищет ближайший except
```

---

### Простейший пример

```python
age = int(input("Введите возраст: "))

if age < 0:
    raise ValueError("Возраст не может быть отрицательным")
```

Здесь нет случайной ошибки. Мы сами создаём правило системы:

> Возраст не может быть отрицательным. Если правило нарушено — генерируется исключение.

---

**Почему это важно?**

Исключения — это не только аварии. Это еще и механизм для:

* защиты системы
* проверки корректности данных
* принудительного соблюдения правил

> Мы переходим от: `Мы ловим ошибки` к `Мы определяем правила`

Это уже архитектурный подход к вашим программам.

---

## Блок 10. Исключения как инструмент логической проверки

До этого момента исключения возникали в результате работы Python: ошибка преобразования, деление на ноль и т.д.

Теперь мы делаем следующий шаг. Мы используем исключения как **механизм проверки правил системы**.

Идея простая:

> Если нарушено правило — программа должна явно сообщить об этом через исключение.

---

### Пример: проверка email и возраста

Требования:

* Email должен содержать символ `@`
* Возраст должен быть числом от 0 до 100

---

### Реализация

```python
try:
    email = input("Введите email: ")
    if "@" not in email:
        raise ValueError("Email должен содержать символ '@'")

    age = int(input("Введите возраст: "))
    if age < 0 or age > 100:
        raise ValueError("Возраст должен быть в диапазоне от 0 до 100")

    print("Данные успешно приняты.")

except ValueError as e:
    print(f"Ошибка ввода данных: {e}")
```

---

### Что здесь происходит?

Разберём по шагам.

```
1. Пользователь вводит данные
2. Программа проверяет правила
3. Если правило нарушено → вызывается raise
4. Создаётся объект ошибки ValueError
5. except перехватывает ошибку
6. Текст ошибки сохраняется в переменной e
7. Мы выводим объединённое сообщение через f-строку
```

---

### Почему используется `raise`?

Рассмотрим альтернативу:

```python
if "@" not in email:
    print("Некорректный email")
```

В этом случае:

* программа просто выводит сообщение
* выполнение продолжается
* система не получает сигнал о критической проблеме

А теперь с `raise`:

```python
if "@" not in email:
    raise ValueError("Email должен содержать символ '@'")
```

Происходит другое поведение:

```
Нарушено правило
        ↓
Создаётся исключение
        ↓
Текущий блок немедленно прерывается
        ↓
Управление передаётся в except
```

> Это принципиально другой уровень контроля.

---

**Исключения** — это не только аварии. Исключения это механизм для:

* фиксации нарушения правил
* остановки некорректного сценария
* централизованной обработки ошибок

> Мы больше не просто реагируем на случайные ошибки. Мы формируем поведение программы.

---

**Исключение** — это сигнал: `"Система не может продолжать работу, потому что нарушено правило."`

Именно поэтому исключения идеально подходят для:

* валидации данных
* проверки диапазонов
* проверки структуры
* контроля состояния программы

```
Правило системы
        ↓
Проверка условия
        ↓
Если условие нарушено → raise
        ↓
except перехватывает исключение
        ↓
Пользователь получает понятное сообщение
```

Программа доллжна:

* Использовать `raise`
* Использовать `except ValueError as e`
* Показать пользователю объединённое сообщение через f-строку

# МОДУЛЬ 4. ИСКЛЮЧЕНИЯ И ФУНКЦИИ

**Архитектура проверок и управление потоком программы**

В предыдущем модуле мы научились создавать исключения через `raise`.

> Теперь важно понять: Как правильно распределить генерацию и обработку исключений между функциями?

Главная идея этого модуля:

- Функция проверки:
    - либо возвращает True,
    - либо генерирует исключение.

- Функция-координатор:
    - объединяет проверки.

---

## Блок 11. Функции проверки (генерация исключений)

Создадим две функции проверки:

* проверка фамилии
* проверка оценки

### Проверка фамилии

```python
def validate_surname(surname):
    if not surname.isalpha():
        raise ValueError("Фамилия должна содержать только буквы")

    return True
```

### Проверка оценки

```python
def validate_grade(grade):
    if grade < 3 or grade > 12:
        raise ValueError("Оценка должна быть в диапазоне от 3 до 12")

    return True
```

---

**Важно подчеркнуть:**

* функции ничего не выводят
* функции ничего не сохраняют
* функции не принимают решений
* они либо подтверждают корректность (`True`)
* либо генерируют исключение

_Это строгая ответственность._

---

## Блок 12. Функция общей проверки

_(Функция-координатор)_

Теперь создадим функцию, которая объединяет проверки.

Функция должна:

* принимает данные
* вызывает функции проверки
* возвращает `True`, если всё корректно
* не занимается выводом сообщений

```python
def validate_student_data(surname, grade):
    try:
        validate_surname(surname)
        validate_grade(grade)
        return True
    except ValueError:
        raise
```

---

### Почему здесь используется `try/except`?

С точки зрения логики, блок `except` здесь просто пробрасывает исключение выше.

Это демонстрирует **важный принцип**:

> Функция может перехватить исключение, но не обязана его обрабатывать. Она может передать его дальше.

> Это называется проброс исключения (`exception propagation`).

---

### Как работает `raise` в этом контексте?

Когда мы пишем `raise` внутри блока `except`, это:

1. Перехватывает исключение, которое возникло в блоке `try`.
2. Сразу пробрасывает его **вверх** по стеку вызовов, то есть к месту, где была вызвана эта функция (в данном случае — к вызову `validate_student_data`).

Таким образом, если в `validate_surname` или `validate_grade` произошло исключение, оно перехватывается в `except ValueError`, но **не обрабатывается**, а просто передается дальше. Это поведение стандартное для Python.

Теперь, если пользователь введёт некорректную фамилию или оценку, например:

```python
validate_student_data("12345", 15)
```

Произойдёт следующее:

* В `validate_surname` возникнет исключение `ValueError`, которое будет перехвачено в `except`.
* `raise` пробросит это исключение обратно на уровень вызова.
* В месте вызова `validate_student_data` нужно будет либо обработать исключение, либо передать дальше.

### Что важно?

* В случае, если исключение **не будет перехвачено на уровне вызова** (например, если в `run_program` или в основной логике нет обработки этого исключения), программа завершится с ошибкой и будет выведено сообщение об ошибке.
* Мы **не обрабатываем** исключение внутри функции `validate_student_data`, а просто передаем его дальше. Это позволяет нам централизованно решать, что делать с ошибками (например, в функции `run_program`).

---

Таким образом, всё работает как ожидается, и не нужно прописывать дополнительную логику для повторной генерации исключения. Python сам пробросит его вверх по стеку вызовов.

---

### Архитектурная схема

```
validate_surname()
validate_grade()
        ↓
validate_student_data()
        ↓
main логика программы
```

Каждый уровень отвечает только за свою часть.

---

## Блок 13. Управляющая функция с ограничением попыток

Теперь добавим главный уровень.

Задача:

* пользователь имеет 3 попытки
* если проверки не проходят — повтор ввода
* после 3 попыток программа завершает работу

```python
MAX_ATTEMPTS = 3

def run_program():
    attempts = 0

    while attempts < MAX_ATTEMPTS:
        try:
            surname = input("Введите фамилию: ")
            grade = int(input("Введите оценку: "))

            validate_student_data(surname, grade)

            print("Данные успешно приняты.")
            return

        except ValueError as e:
            attempts += 1
            print(f"Ошибка: {e}")
            print(f"Осталось попыток: {MAX_ATTEMPTS - attempts}")

    print("Превышено допустимое количество попыток.")
```

**Запуск**:

```python
run_program()
```

---

### Полная схема программы

```
Ввод данных
        ↓
validate_student_data()
        ↓
    validate_surname()
    validate_grade()
        ↓
Если ошибка → исключение
        ↓
run_program() перехватывает
        ↓
Уменьшает количество попыток
        ↓
Либо повтор, либо завершение
```

---

### Важный архитектурный вывод

Мы получили многоуровневую структуру:

| Уровень               | Ответственность      |
| --------------------- | -------------------- |
| validate_*            | Правила              |
| validate_student_data | Объединение правил   |
| run_program           | Управление процессом |

Исключения стали механизмом управления логикой всей программы.

## Блок 14. Блок `else`

### Когда выполняется `else`?

Блок `else` выполняется **только если в `try` не возникло исключений**.

Схема:

```
try:
    основной код
except:
    если произошла ошибка
else:
    если ошибки НЕ было
```

---

**Пример**:

```python
try:
    number = int(input("Введите число: "))
except ValueError:
    print("Ошибка ввода.")
else:
    print("Число успешно преобразовано.")
```

Логика работы:

- Если int() вызвал ошибку → except
- Если int() прошёл успешно → else


---

Блок `else` позволяет:

* отделить основной код от реакции на успех
* сделать структуру более читаемой
* не смешивать логику обработки ошибок и обычную логику

Без `else` пришлось бы писать всё внутри `try`, что ухудшает структуру.

---

### Когда `else` действительно полезен?

1. Когда основной код должен выполняться **только если ошибок не было**
2. Когда нужно явно показать «ветку успеха»
3. Когда `try` содержит только потенциально опасные операции

---

## Блок 15. Блок `finally`

Теперь более важный блок.

Блок `finally` выполняется **всегда**:

* произошла ошибка
* не произошла ошибка
* был `return`
* был `raise`

---

### Блок `finally` особенно важен при работе с файлами

При открытии файла мы получаем ресурс:

```python
file = open("data.txt", "r")
```

Этот ресурс нужно закрыть:

```python
file.close()
```

Мы уже с вами знаем, что если файл не закрыть:

* возможна утечка ресурсов
* файл может остаться заблокированным
* программа может вести себя нестабильно

И вот здесь появляется `finally`.

---

### Пример с файлом (CSV как простой текст)

```python
file = None

try:
    file = open("students.csv", "r")
    content = file.read()
except FileNotFoundError:
    print("Файл не найден.")
else:
    print("Файл успешно прочитан.")
finally:
    if file is not None:
        file.close()
```

---

**Что происходит?**

* Пытаемся открыть файл
* Если файла нет → except
* Если файл открыт успешно → else
* В любом случае → finally закрывает файл

Даже если внутри `try` возникнет другая ошибка, файл всё равно будет закрыт.

Это и есть главная ценность `finally`.

---

### Визуальная схема выполнения

```
        TRY
          ↓
  ┌───────────────┐
  │    Ошибка?    │
  └───────────────┘
     ↓           ↓
   Да            Нет
     ↓           ↓
  except        else
       \        /
        \      /
         finally
```

`finally` всегда выполняется в конце.

---

### Можно ли показать это на JSON?

Да. Логика идентичная.

Пример:

```python
import json

file = None

try:
    file = open("config.json", "r")
    data = json.load(file)
except FileNotFoundError:
    print("Файл конфигурации не найден.")
except json.JSONDecodeError:
    print("Файл повреждён.")
else:
    print("Конфигурация успешно загружена.")
finally:
    if file is not None:
        file.close()
```

Если файл повреждён — `json.load()` вызовет исключение.
Но `finally` всё равно выполнится.

_Эту логику можно применять для любых файлов._

---

## Блок 16. Еще немного о блоке `finally`

Часто создаётся впечатление, что `finally` нужен только для:

* закрытия файлов
* освобождения ресурсов

Но его назначение шире.

Если `finally` — это блок, который выполняется **всегда**, независимо от результата выполнения программы реализовать отдельную логику, значит это делает его **идеальным местом для**:

* фиксации завершения операции
* очистки временных данных
* сброса состояния
* вывода финального статуса
* служебных сообщений

---

### Пример: простая дополнительная логика в `finally`

```python
file = None

try:
    file = open("students.csv", "r")
    content = file.read()
except FileNotFoundError:
    print("Файл не найден.")
else:
    print("Файл успешно прочитан.")
finally:
    if file is not None:
        file.close()
    print("Операция завершена.")
```

Обрати внимание:

Сообщение `"Операция завершена."` выводится:

* если файл найден
* если файл не найден
* если внутри блока произошла другая ошибка

---

_***Важно**: этот пример был разработан только для визуальной и понятной демонстрации работы блока `finally`. В реальных проектах работа с файлами в 90% случиях происходит с помощью конструкции контекстного менеджера `with`._

---

### Где `finally` применяется в реальных программах?

1. Освобождение ресурсов (файлы, соединения)
2. Разблокировка объектов
3. Очистка временных переменных
4. Вывод итогового статуса операции
5. Сброс счётчиков или флагов
6. Завершение транзакции

---

### Частые исключения при работе с файлами

При работе с файлами ошибки возникают достаточно часто. Важно не ловить «всё подряд», а понимать, какие именно исключения возможны.

| Исключение             | Когда возникает                                  | Пример ситуации                                 |
| ---------------------- | ------------------------------------------------ | ----------------------------------------------- |
| `FileNotFoundError`    | Файл не существует по указанному пути            | Опечатка в имени файла                          |
| `PermissionError`      | Недостаточно прав доступа                        | Попытка открыть системный файл                  |
| `IsADirectoryError`    | Вместо файла указана папка                       | Попытка `open("folder", "r")`                   |
| `NotADirectoryError`   | В пути указана часть, которая не является папкой | Неверная структура пути                         |
| `UnicodeDecodeError`   | Ошибка декодирования текста                      | Файл открыт в неверной кодировке                |
| `ValueError`           | Неверная логика работы с файлом                  | Попытка чтения закрытого файла                  |
| `json.JSONDecodeError` | Некорректный формат JSON                         | Повреждённый JSON-файл                          |
| `csv.Error`            | Ошибка структуры CSV                             | Некорректное количество колонок                 |
| `OSError`              | Общая ошибка операционной системы                | Повреждённый диск, проблемы с файловой системой |

## Блок 17. Зачем нужно логирование

До этого момента мы:

* обрабатывали исключения;
* пробрасывали ошибки вверх по стеку вызовов;
* использовали `finally`;
* работали с файлами, где часто возникают сбои.

Но возникает практический вопрос:

> Что происходит с ошибкой после того, как пользователь закрыл программу?

* Если программа завершилась с ошибкой — вы её увидели.
* Если программа работает у пользователя — вы уже ничего не видите.

Именно здесь появляется логирование.

---

### Что такое логирование

**Логирование** — это систематическая запись информации о работе программы.

> Лог — это журнал событий программы.

Программа может записывать:

* этапы своей работы
* диагностическую информацию
* предупреждения
* ошибки
* критические сбои

---

> `print()` — это не логирование

Для начинающих разработчиков использовать `print()` для отладки и контроля работы программы - это нормальное построение логики.

Но с увеличением кода и усложнением логики ваших программ просто `print()` будет уже не достаточно.

Сравним.

**Использование `print()`**

* выводит информацию только в консоль
* исчезает после завершения программы
* не разделяет уровни важности
* не подходит для production-кода

**Использование Логирование**

* записывает данные в файл
* хранит историю работы
* разделяет сообщения по уровням важности
* позволяет фильтровать сообщения
* используется в реальных проектах

---

**Когда логирование особенно необходимо**

В реальных проектах:

* серверные приложения
* Telegram-боты
* фоновые задачи
* веб-сайты
* API
* финансовые системы

Если пользователь сообщает: `"У меня ничего не работает"`

Вы открываете лог-файл и видите:

```
2026-02-15 12:04:22 ERROR FileNotFoundError: data.csv not found
```

**И проблема становится очевидной**.

---

### Уровни важности (Severity Levels)

Логирование — это не просто запись текста.

Это запись **со степенью важности**.

Основные уровни:

| Уровень  | Назначение                                      |
| -------- | ----------------------------------------------- |
| DEBUG    | Подробная отладочная информация                 |
| INFO     | Обычные события работы программы                |
| WARNING  | Подозрительная ситуация                         |
| ERROR    | Ошибка, но программа продолжает работу          |
| CRITICAL | Критическая ошибка, программа может завершиться |

---

### Визуальная иерархия уровней

```
CRITICAL  ↑ Самый высокий уровень
ERROR
WARNING
INFO
DEBUG     ↓ Самый низкий уровень
```

Если установлен уровень `WARNING`, то:

* DEBUG → не записывается
* INFO → не записывается
* WARNING → записывается
* ERROR → записывается
* CRITICAL → записывается

Это называется **фильтрация по уровню**.

---

### Где используется чаще всего

В большинстве **реальных проектов** активно используются:

* INFO
* WARNING
* ERROR

`DEBUG` используется во время разработки.

`CRITICAL` используется редко — только для серьёзных аварийных ситуаций.

## Блок 18. Библиотека `logging`

В Python стандартной библиотекой для логирования является модуль `logging`.

Он встроен в язык и не требует установки.

---

### Базовая настройка

Самый простой способ настроить логирование:

```python
import logging

logging.basicConfig(
    filename="app.log",
    level=logging.INFO,
    encoding="utf-8"
)

logging.info("Программа запущена")
```

После запуска появится файл:

```
app.log
```

---

### Что делает `basicConfig`

Параметры:

* `filename` — файл для записи логов
* `level` — минимальный уровень важности
* `encoding` — кодировка файла

Так же в базовую настройку можно добавить:

* `format` — формат строки (если не указан — используется стандартный)
* `filemode` - способ открытия файла (по умолчанию "a")

---

### Пример с форматом

```python
logging.basicConfig(
    filename="app.log",
    level=logging.INFO,
    encoding="utf-8",
    format="%(asctime)s - %(levelname)s - %(message)s"
)
```

Теперь лог будет выглядеть так:

```
2026-02-15 14:32:10,123 - INFO - Программа запущена
```

---

### Как работают уровни

Каждому уровню соответствует метод:

```python
logging.debug("Отладка")
logging.info("Информация")
```

## Блок 19. Практика: логирование работы с файлом

Напишим программу, которая:

* читает файл
* обрабатывает данные
* обрабатывает ошибки
* логирует поведение

---

### Архитектура программы

```
main()
  ↓
read_file()
  ↓
process_data()
  ↓
write_result()
```

Ошибки могут возникнуть:

* при чтении
* при обработке
* при записи

---

### Пример программы с логированием

```python
import logging


logging.basicConfig(
    filename="program.log",
    level=logging.INFO,
    encoding="utf-8",
    format="%(asctime)s - %(levelname)s - %(message)s"
)


def read_file(filename):
    """
    Функция читает данные
    """
    logging.info(f"Попытка открыть файл {filename}")
    with open(filename, "r", encoding="utf-8") as f:
        data = f.readlines()
    logging.info("Файл успешно прочитан")
    return data


def process_data(lines):
    """
    Функция обрабатывает данные
    """
    logging.info("Начата обработка данных")
    processed = []

    for line in lines:
        line = line.strip()
        if not line:
            logging.warning("Обнаружена пустая строка")
            continue

        processed.append(line.upper())

    logging.info("Обработка завершена")
    return processed


def write_result(filename, data):
    """
    Функция записывает данные
    """
    logging.info(f"Запись результата в файл {filename}")
    with open(filename, "w", encoding="utf-8") as f:
        for line in data:
            f.write(line + "\n")
    logging.info("Результат успешно записан")


def main():
    try:
        lines = read_file("input.txt")
        processed = process_data(lines)
        write_result("output.txt", processed)

    except FileNotFoundError as e:
        logging.error(f"Файл не найден: {e}")

    except PermissionError as e:
        logging.error(f"Нет доступа к файлу: {e}")

    except Exception as e:
        logging.critical(f"Необработанная ошибка: {e}")

    finally:
        logging.info("Работа программы завершена")


if __name__ == "__main__":
    main()
```

---

**В этой программе**:

* Каждое значимое действие логируется
* Предупреждения фиксируются отдельно
* Ошибки фиксируются отдельно
* Критические ошибки выделяются отдельно
* Завершение программы всегда записывается через `finally`
